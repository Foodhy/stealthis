Architecture

Next.js App Router

Project structure with nested layouts, server/client components, route handlers, and middleware

Folder Structure
app/ ├── layout.tsx (Root layout) ├── page.tsx (Home /) ├── loading.tsx (Loading UI) ├── error.tsx (Error boundary) ├── not-found.tsx (404 page) ├── middleware.ts (Edge middleware) ├── api/ │ └── route.ts (Route handler) ├── dashboard/ │ ├── layout.tsx (Nested layout) │ ├── page.tsx (/dashboard) │ └── settings/ │ └── page.tsx (/dashboard/settings) └── (auth)/ ├── login/page.tsx (Route group) └── register/page.tsx
Key Files
layout.tsx Persistent UI shell; root layout replaces _app and _document
page.tsx Makes a route segment publicly accessible
loading.tsx Instant loading state via React Suspense boundaries
error.tsx Error boundary scoped to the segment and its children
route.ts API endpoint with GET, POST, PUT, DELETE handlers
middleware.ts Intercepts requests for auth, redirects, and rewrites
Routing Conventions

Nested Layouts

Layouts wrap child routes automatically. A dashboard layout persists across all /dashboard/* pages without re-rendering the root.

app/dashboard/layout.tsx

Route Groups

Parenthesized folders like (auth) organize routes without affecting the URL. login/page.tsx renders at /login.

(auth)/login/page.tsx
Server vs Client

Server Components

Default in App Router. Zero JS bundle cost. Direct access to databases, filesystem, and secrets. No useState or event handlers.

Default behavior

Client Components

Add "use client" directive for interactivity. Required for useState, useEffect, onClick, and browser APIs.

"use client"
