Architecture

Micro-Frontend (Module Federation)

Independent host/remote applications sharing dependencies at runtime via Webpack 5

Folder Structure
monorepo/ ├── host-app/ (Shell application) │ ├── webpack.config.js (ModuleFederationPlugin) │ ├── src/ │ │ ├── App.tsx (Lazy-loads remotes) │ │ └── bootstrap.tsx (Async boundary) │ └── package.json ├── remote-dashboard/ (Independent micro-app) │ ├── webpack.config.js (Exposes modules) │ ├── src/ │ │ ├── Dashboard.tsx (Exposed component) │ │ └── bootstrap.tsx │ └── package.json ├── remote-settings/ (Another micro-app) │ └── ... └── shared/ (Shared types & utils)

Shared Dependencies

Version Negotiation

When host and remotes declare the same shared dependency, Module Federation loads only one copy using semver rules to pick the best version.

shared config

Async Boundary

Each app uses a bootstrap.tsx that dynamically imports the root. This ensures federated modules are resolved before React mounts.

bootstrap.tsx
Independent Deployment

Separate CI/CD

Each micro-app has its own pipeline and deployment. Updating a remote does not require rebuilding or redeploying the host.

Independent deploys

Runtime Integration

The host fetches each remote's remoteEntry.js manifest at runtime and dynamically loads the requested modules on demand.

remoteEntry.js
When to Use

Good Fit

Multiple teams shipping independently, large apps needing runtime integration, different framework versions coexisting.

Scale

Avoid When

Small teams, simple applications, or when build-time composition (monorepo imports) is sufficient. The operational overhead is significant.

Complexity cost
